圣杯和双飞翼布局

Posted on 2019-09-16

花点时间整理下圣杯和双飞翼布局

主要是解决左右两边定长 中间自适应的问题。
当然在现在我觉得如果浏览器环境允许的话 使用Flex 最好代码量也少

先写个总结 圣杯/双飞翼
个人感觉还是圣杯语感比较好一些，因为它把左右两边div进行包裹起来写的。

  • 主要考察的问题 margin-left/right在float环境下的影响 margin的负值会进行覆盖前面的dom元素

  • 主要解决的问题

    • 左右两边固定长度，中间的div进行可适应。
    • 中间部分在dom结构上优先 以便于解析
    • 允许三列中的任意一列为最高列
    • 需要用额外的div

圣杯模式
dom结构信息

.container{
padding: 0 100px;
}
.middle {
width: 100%;
}
.left: {
width: 100px;
margin-left: -100%; // 这里margin对应的百分比是自己的父元素 所以这里就是 middle的宽度咯 负数就是将这个dom往左移动 100%的父宽度。会盖住middle
// 然后进行定位处理
position: relative;
left: -100px; //向左移动100px
}
.right: {
width: 100px;
margin-right: -100px; //同理
}

<div>
    <div class="header"></div>
    <div class="container">
        <div class="middle">middle</div>
    </div>
    <div class="left">left</div>
    <div class="right">right</div>
    <div clas="footer"></div>
</div>

css部分 都是左浮动， 我大概写一下主要区别的内容
.container{
float: left;
width: 100%;
}
.middle{
margin: 0 100px;
}
.left{
width: 100px;
margin-left: -100%; //由于父元素就是100%的长度 所以 双飞翼用不到定位的操作
}
.right{
width: 100px;
margin-left: -100px;
}

浏览器渲染一个html 都经历了什么？

Posted on 2019-09-15

浏览器渲染一个html都发生了些什么？

废话不多说 直接上图

  1. 解析代码
    • 将html => dom tree
    • 将css => cssDom styletree 这里的css样式都是全部经过计算 平常写的都是相对值 需要经过继承、计算等。
  2. 合并树(render tree)
    • 从dom根节点进行遍历 和cssdom进行合并 生成渲染树(render tree)。
  3. layout css提供了部分布局信息,但浏览器想要的是相对于设备视窗的确切信息和大小，仅仅靠render tree不能全部描述。缺少节点之间的关系等。如果修改了布局 就会触发layer进行重新计算
  4. painting 遍历渲染树，进行每一个节点的绘制

最后有一个composite这个存在争议

当我们的样式进行更新了 发生了什么呢？

  1. 通过js来动态更新了样式
  2. 判断是不是布局的修改
    • 如果是修改了布局 js => style => render => layout => painting => compoisition
    • 如果修改的是样式不是有关于布局的 js=> style => render => painting=> composition 不会进行触发layout的计算

中秋晚

Posted on 2019-09-14

你不知道的javascript(中) 关于promise 和Generator的相关内容

日常中我所遇到的问题和书中的共识？ 异步的问题
  • 遇到的问题
    1. 调用过早
    2. 调用过晚
    3. 回调被多次调用
    4. 一次不调(环境等因素 参数成功没有调用)
    5. 异常被吞了
历史原因

由于依赖第三方库 在早期这并不定是安全的 我们给予某第三方库，如果调用成功，执行我们的回调。 可能它内部的问题导致生产环境下异常 这是非常严肃的。

解决办法
ES6 提供了Promise来解决上面的额问题

  1. 调用过早？ 这个是不存在的 实际上then的时候是异步的 我手写过模拟promise 实际上then的时候如果在回调之前只是将方法push到数组里，当结果实行后 然后forEach数组的内容。
  2. 调用过晚？ 不存在这种情况，并且实际上可以设置超时时间等辅助的方法
  3. 调用次数？ 状态只会从pending => fulfilled / rejected 且不会被改变 只能一次 俗称单一性
  4. 一次不掉？ 只有resolve/reject两种可能
  5. 异常 结果是放置在reject上进行

另几个API比较重要
Promise.all / race
分别是 门栏和门闩的区别。 all是所有的promise都执行了且是并发的的情况才会被then race类似于抢占式 有一个改变状态就改变了状态触发了then

Promise也有自身的缺陷

  1. 顺序错误的处理问题，分不清是哪个环节的问题
  2. 单一值 如果要多值的话只能再封装 不过可以使用promise.all来进行转化
  3. 单决值 只有从pending=> fulfilled/rejected 两种状态并且回不去 有时候比较鸡肋
  4. 无法取消
  5. 性能上肯定有损耗一些的 但为了上面的5种问题的解决方案 可以忍
